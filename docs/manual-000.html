1. Propósito

Luis G. Analysis nace para convertir conocimiento disperso en un sistema organizado, claro y reutilizable. Su función no es acumular notas sin orden, sino construir una biblioteca de referencia que pueda consultarse meses o años después.

La biblioteca debe reunir manuales, guías, documentos técnicos, lecciones aprendidas, listas de verificación, recursos y versiones descargables de los temas importantes para el aprendizaje, el trabajo, los proyectos personales y la toma de decisiones.

2. Filosofía central

La filosofía principal del proyecto se resume en una frase:

La memoria falla. Los sistemas permanecen.

La memoria personal es limitada. Una conversación se pierde, una idea se olvida y una solución puede quedar enterrada entre mensajes. Un sistema bien documentado permite conservar lo aprendido, repetir procesos, evitar errores y mejorar con cada versión.

3. Principios editoriales

  • Claridad: todo documento debe poder entenderse sin esfuerzo innecesario.
  • Utilidad práctica: el contenido debe ayudar a decidir, ejecutar, aprender o consultar.
  • Orden: cada tema debe tener una estructura lógica y fácil de navegar.
  • Precisión: evitar afirmaciones dudosas, exageradas o sin contexto.
  • Consistencia: usar términos, formatos y criterios similares en toda la biblioteca.
  • Actualización: los documentos deben poder mejorar por versiones.
  • Seguridad: no publicar datos sensibles, credenciales ni información privada.

7. Sistema de versionado

Los documentos no necesitan nacer perfectos. Deben nacer claros y mejorar por versiones.

  • V0.1: primera versión útil.
  • V0.2: ampliación estructural.
  • V0.5: documento sólido pero todavía en revisión.
  • V1.0: primera versión estable.
  • V1.1 o superior: mejoras, correcciones y actualizaciones.

8. Flujo de trabajo editorial

  1. Definir el objetivo del documento.
  2. Crear una estructura base.
  3. Redactar una versión inicial útil.
  4. Revisar claridad, ortografía y coherencia.
  5. Publicar versión web.
  6. Crear versión PDF cuando el contenido esté suficientemente estable.
  7. Actualizar el índice de la biblioteca y la página de descargas si corresponde.
  8. Registrar cambios en el historial de versiones.

9. Criterios de calidad

Un documento se considera aceptable cuando cumple estos criterios:

  • El tema está claramente definido.
  • La estructura permite encontrar información con facilidad.
  • El texto tiene buena ortografía y acentuación.
  • Las explicaciones son suficientes para entender y aplicar.
  • No contiene relleno innecesario.
  • Incluye ejemplos, pasos o criterios cuando aportan valor.
  • No contiene información privada o sensible.

10. Seguridad y privacidad

Regla obligatoria: no publicar contraseñas, claves API, tokens, datos bancarios, documentos personales, rutas privadas sensibles, configuraciones críticas ni información que pueda comprometer cuentas, servidores o proyectos.

Cuando un documento requiera explicar procesos técnicos, se deberán usar ejemplos genéricos, rutas ficticias o datos anonimizados.

11. Lista de verificación antes de publicar

  • El título es claro.
  • La versión está indicada.
  • La introducción explica el propósito.
  • La estructura está ordenada.
  • La ortografía fue revisada.
  • Los enlaces funcionan.
  • No hay datos sensibles.
  • El documento aporta valor práctico.
  • El historial de versiones está actualizado.
